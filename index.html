Figma Make

AI-powered UI generation built directly into Figma — describe what you want, watch it appear.

How it works

Prompt → Prototype

Figma Make takes a natural language prompt and generates a complete, editable Figma design — frames, components, variants, auto-layout and all.

What it understands

Design intent & context

It infers layout patterns, spacing hierarchies, component structures, and applies them using Figma's native primitives — not just flat art.

Where to access

Figma Sidebar

Available in the right panel under Make. Works in Design mode. Can reference selected components, styles from your local file or libraries.

Use Cases

01
Rapid Wireframe Generation
Exploration ▼

Jump from a blank canvas to a structured wireframe in seconds. Great for early-stage exploration when the layout is still undefined.

  • Open Figma Make panel
  • Describe the screen type, content blocks, and key user actions
  • Accept or iterate with follow-up prompts ("make the header taller", "add a sidebar nav")
  • Convert generated layers into proper auto-layout components
Example prompt
"Create a SaaS dashboard wireframe with a sidebar navigation, a KPI cards row at the top, a main chart area, and a recent activity feed on the right."
02
Component Variant Generation
Systems ▼

Generate full component sets with all states and variants — buttons, form inputs, cards, modals — that align to your existing design system tokens.

  • Select an existing component or provide a description
  • Prompt Figma Make to generate all required variants
  • Review variants and adjust naming/property structure
  • Publish to your team library
Example prompt
"Generate a Button component with variants: Size (sm/md/lg), State (default/hover/disabled/loading), Type (primary/secondary/ghost/destructive). Use our existing color tokens."
03
UI Copy & Content Population
Content ▼

Replace Lorem Ipsum with contextual, realistic placeholder content that matches your product domain — names, numbers, dates, microcopy.

  • Select frames or components with placeholder text
  • Prompt to populate with contextually relevant content
  • Useful for user testing sessions and stakeholder reviews
Example prompt
"Fill the user card components with realistic data: Vietnamese names, email addresses, realistic last login dates, and varied subscription plan labels (Free, Pro, Enterprise)."
04
Responsive Layout Adaptation
Layout ▼

Adapt desktop designs to tablet and mobile layouts without manually rebuilding every frame. Figma Make understands responsive layout intent.

  • Select the desktop frame you want to adapt
  • Prompt for mobile/tablet version with specific breakpoint constraints
  • Refine collapsed navigation, stacked elements, touch targets
Example prompt
"Create a mobile (375px) version of this dashboard. Collapse the sidebar into a bottom tab bar, stack the KPI cards into a horizontal scroll row, and hide the right panel behind a floating button."
05
Design System Documentation
Documentation ▼

Auto-generate documentation frames for components — usage guidelines, do/don't examples, anatomy annotations.

  • Select a component
  • Ask Figma Make to generate a documentation page layout
  • Include usage examples, anti-patterns, accessibility notes
Example prompt
"Create a component documentation page for the Modal component. Include: anatomy diagram with labels, usage do/don't examples (2 each), size variants showcase, and a note about keyboard accessibility."

Claude via Figma MCP

Give Claude direct access to your Figma files — read, create, edit, and audit designs through natural language in Claude.ai.

What is MCP?

Model Context Protocol

MCP is an open protocol that lets AI models connect to external tools. The Figma MCP lets Claude read and manipulate your Figma files directly.

Setup

Connect in Claude.ai

Enable the Figma connector in Claude.ai settings → Integrations. Authorize with your Figma account. Claude can now access your files.

What Claude can do

Read, Create, Edit, Audit

Read file structure and component trees, create new frames and components, update properties, run audits, and export structured summaries.

Use Cases

01
Design System Audit
Audit ▼

Claude reads the entire Figma file and produces a detailed report: inconsistent color usage, hardcoded values that should be tokens, missing component descriptions, and accessibility flags.

  • Share the Figma file URL with Claude
  • Ask for a design system health check
  • Claude lists: unstyed layers, non-token colors, missing descriptions, naming inconsistencies
  • Ask Claude to fix specific issues in bulk
Example prompt
"Audit the design system file at [URL]. List all components missing descriptions, any color styles that appear to be duplicates (similar values), and components that don't follow the naming convention [Category/Variant/State]."
02
Bulk Rename & Restructure
Automation ▼

Rename hundreds of layers, components, or pages at once based on a pattern or convention — without clicking through each one manually.

  • Describe the renaming pattern to Claude
  • Claude identifies all matching nodes via the MCP
  • Applies renames in batch
  • Claude confirms the changes made
Example prompt
"In my Figma file, rename all button components to follow the convention: Button/[Size]/[Type]/[State]. Current naming is inconsistent (btn-primary, PrimaryButton, button_primary etc.). Apply the new convention across all 47 instances."
03
Dev Handoff Documentation
Handoff ▼

Claude reads your Figma designs and produces structured developer documentation: spacing values, color tokens, component props, interaction notes — ready to paste into Notion or Linear.

  • Point Claude to a specific screen or component
  • Ask for a handoff spec in your preferred format
  • Claude extracts every measurable value from the design
  • Outputs structured markdown or JSON for dev consumption
Example prompt
"Read the 'Checkout Flow' page in my Figma file. Generate a dev handoff document in markdown format. Include: all spacing values, color tokens used, component props for each interactive element, and any noted interaction behaviors from the comments."
04
Cross-Screen Consistency Check
QA ▼

Claude compares multiple screens to find inconsistencies in spacing, typography, component usage, and visual hierarchy — like having a QA partner who's read every pixel.

  • Share multiple screen frames or the full page
  • Ask Claude to compare header heights, navigation patterns, button placements
  • Get a structured inconsistency report
  • Prioritize and fix before development
Example prompt
"Compare all 12 screens in my 'Onboarding Flow' Figma page. Identify any inconsistencies in: primary CTA button position, heading font sizes, horizontal padding on content areas, and icon style (outline vs filled)."
05
Token Architecture Planning
Tokens ▼

Claude reads your existing styles and variables, analyzes usage patterns, then helps you design a proper tiered token architecture (global → semantic → component).

  • Ask Claude to browse your current Figma variables/styles
  • Claude identifies the current structure and gaps
  • Proposes a semantic token layer mapping
  • Can create the new variable collections in Figma directly
Example prompt
"Browse my Figma variables. I currently have only raw color values. Design a 3-tier token system: global (primitives), semantic (contextual aliases like color/background/primary), and component (button/background/default). Then create the semantic collection in my file."
06
Accessibility Audit
Accessibility ▼

Claude runs a WCAG-aware accessibility review — checking contrast ratios, tap target sizes, text sizing, and flagging patterns that would fail accessibility requirements.

  • Share the file or specific screen
  • Claude uses figma_lint_design to run checks
  • Returns a list of violations by WCAG level (A, AA, AAA)
  • Suggests specific fixes for each issue
Example prompt
"Run a WCAG AA accessibility audit on the 'Product Page' in my Figma file. List all contrast ratio failures, any text under 16px that isn't bold, interactive elements smaller than 44×44px, and missing focus states."

Live Demo Script

End-to-end walkthrough: from blank canvas to dev-ready design in 20 minutes.

Demo A · Figma Make · 10 min

Build a Settings Page from Scratch

Demonstrate how quickly a structured, production-quality screen can emerge from a single prompt — and how to iterate with follow-ups.

1.
Open Figma Make panel
Show the audience where it lives. Mention it's available on any file, no setup needed.
2.
Enter initial prompt
"Design a Settings page for a B2B SaaS product. Sections: Profile, Security, Notifications, Billing, Team Members. Use a two-column layout with a sidebar nav on the left."
3.
Review and narrate what was generated
Point out auto-layout, component usage, consistent spacing. Mention what's great AND what needs adjustment — show the real workflow, not perfection.
4.
Iterate with a follow-up prompt
"Change the sidebar to use icon + label pairs. Add a Danger Zone section at the bottom of the Profile tab with a Delete Account action in red."
5.
Generate the mobile version
"Create a mobile (390px) version. Replace the sidebar with a segmented control tab bar at the top. Stack all form rows vertically."
Demo B · Claude MCP · 10 min

Audit & Improve an Existing Design File

Show Claude's ability to read, reason about, and modify a real Figma file — demonstrating the power differential from manual work.

1.
Show the Figma connector in Claude.ai
Settings → Integrations → Figma. Show it's connected. This is the setup moment — do it beforehand but walk through it briefly.
2.
Run a design system audit
"Open my Figma file [URL]. Audit the design system page. Tell me: how many components have missing descriptions? Are there any duplicate color styles? Which components use hardcoded colors instead of tokens?"
3.
Show Claude's output — point out structure
Claude returns a structured list. Show that this would take 2–3 hours manually. Here it took 30 seconds. This is the "wow moment" for the audience.
4.
Ask Claude to fix one category
"Add descriptions to all 12 components that are missing them. Use this format: '[Component name] — [one sentence on purpose and key variants]'. Apply directly to the Figma file."
5.
Switch back to Figma to show the changes live
Flip to Figma and show the component descriptions have been updated. This live verification is the closing moment — powerful and concrete.

Best Practices & Tips

What we've learned from working with these tools daily — what works, what doesn't.

✦ Figma Make

🎯

Be specific about layout constraints

Say "two-column, 320px sidebar, 1fr main" rather than "two column layout." Precision in layout parameters yields far better results.

📐

Reference your design system explicitly

Mention your token names or existing component names in the prompt. Figma Make will try to use them if your library is connected to the file.

🔁

Iterate incrementally

Don't try to get everything right in one prompt. Start with structure, then refine typography, then add states. Small focused prompts beat one giant one.

🧹

Always clean up layers after generation

Rename layers, convert auto-groups to proper auto-layout frames, and replace any unstyled colors with tokens before publishing to a project file.

⬡ Claude via Figma MCP

🔗

Always share the full Figma URL

Include the file URL with the specific page node ID fragment (#page-id) so Claude opens the right page directly. This saves one round-trip query.

📋

Ask for a summary before bulk operations

"First, tell me what you find before making any changes." Review Claude's plan before it executes. Easier than undoing 200 renames.

⚡

Combine audit + fix in one session

Instead of two separate chats, ask Claude to audit, present findings, and then say "fix items 1, 3, and 5." Maintains full context throughout.

📤

Use for handoff output, not handoff replacement

Claude generates handoff specs from design — but engineers still need to verify measurements. Use it to cut documentation time by 70%, not eliminate review.
